Horae 配置檔
導出目前所有開關、僅限主角設定、客製化提示詞、RPG配置等為資料（不含自動摘要與向量記憶設定）。角色卡作者可將配置隨卡分享，玩家導入後一鍵到位。
基本設定
關閉後頂部圖示完全隱藏，不留空位。可透過擴展面板（積木圖示）中的「Horae 時光記憶」欄位重新開啟。
訊息面板佔聊天區域的寬度百分比（50-100）。寬屏美化可調低避免溢出。右偏移可微調面板水準位置。
外觀設定
選擇內置主題或導入的客製化美化。導入美化資料（.json）後會出現在下拉列表中。
客製化CSS
輸入客製化CSS覆蓋插件樣式。適合進階用戶微調配色。修改後切換焦點即生效。
高級設定
控制發送給AI的"一般"層級劇情軌跡數量：
0 = 不發送一般事件（重要/關鍵照發）
15 = 發送最近15條一般事件（預設）
數字越大發送越多，Token消耗越高
Horae狀態資訊插入到AI提示詞的哪個位置：
0 = 最末尾（緊貼最新訊息）
1 = 倒數第二（推薦，給AI留思考空間）
數字越大越靠前
客製化表格
建立Excel風格表格，讓AI按你的需求填寫特定資訊。
長按表頭可添加/刪除行列。每個表格可設定提示詞指導AI如何填寫。
本地表格跟隨目前對話保存；全域表格所有對話共享。點擊表格名稱前的圖示可切換。
🔒 鎖定行列可防止AI修改特定內容。
發送給AI的內容
控制哪些資訊會隨提示詞發送給AI。關閉可節省Token，但AI將無法"記住"這些內容。
目前注入 Token：計算中...
歷史事件摘要、相對時間計算。關閉後AI無法知道"幾天前"發生了什麼。
在場角色、服裝描述、好感度、角色資訊。關閉後AI可能描寫錯誤的服裝。
目前持有的物品及位置。關閉後AI可能"憑空"取出不存在的物品。
記錄地點的固定物理特徵，保持跨回合場景描寫一致性。開啟後AI會在首次到達新地點時記錄特徵描述。
追蹤角色間的關係變化（朋友、戀人、敵對等），AI僅在關係變化時輸出。關閉後不生成rel:標籤，不注入關係網路提示詞。
追蹤在場角色的情緒/心理狀態變化，幫助AI保持角色情緒連貫性。AI僅在情緒明顯變化時輸出。
啟用後，AI 會在 <horaerpg> 標籤中輸出角色屬性條（HP/MP/SP）和技能變化。關閉時不注入提示詞、不消耗 Token。
適用於「USER自己描寫行為/動作，AI不再轉述USER部分」的寫法。開啟後，AI回覆時會將上一條USER訊息的內容也納入本條標籤的結算，確保USER的行為不遺漏。
啟用後，底部面板會出現「標記為番外」按鈕。被標記的訊息不會被Horae追蹤、不參與自動摘要，適合在主線中穿插if/番外/小劇場。
自動摘要 & 隱藏
超過緩衝上限後自動將舊事件交給AI壓縮為摘要，節省Token。
壓縮後的摘要可在時間線中切換回原始事件查看。
最近N條訊息絕對不會被摘要。確保近期劇情細節完整保留。
超過此上限時觸發自動壓縮。
舉例：假設保留=10，緩衝上限=20（樓層模式）
→ 你聊到第 50 樓時：第 41-50 樓絕對保留不摘要
→ 第 1-40 樓中，未被摘要的部分如果 超過 20 樓就觸發一次自動摘要
→ 每次摘要按照下方設定的批次上限（訊息條數或Token數）處理一批，剩餘的留到下一輪
簡單理解：保留 + 緩衝 = 多少樓之後才開始摘要
每次自動摘要最多處理多少條訊息。舊檔案首次啟用時會分多輪逐步消化。
每次摘要送入的文字 Token 上限，超出則截斷至此值。
參考各模型上下文視窗（建議設為 30%~50%）：
Claude ≈ 80K~200K · GPT-4o ≈ 80K~128K · Gemini ≈ 100K~1000K · DeepSeek ≈ 64K
填入一個 OpenAI 相容端點，摘要生成將直接請求該端點，不佔用酒館主連接。
OpenAI 相容端點基礎地址，如 https://api.openai.com/v1
填好地址和密鑰後點擊 拉取。低成本模型即可。
填寫完成後，自動摘要將透過獨立 HTTP 請求並行生成，不影響酒館主 API。
支援所有 OpenAI 相容格式的端點（OpenAI、DeepSeek、OpenRouter 等）。
連接測試僅查詢模型列表，不消耗生成次數。
向量記憶 (Beta)
對 Horae 標籤數據做向量檢索，自動召回相關歷史片段注入給 AI。
本地模式：下載約 25MB 量化模型到瀏覽器快取，佔用 50-80MB 主記憶體，⚠ 2GB 以下手機或雲酒館不建議使用，請務必確保有配置—max-old-space-size解除Node.js執行主記憶體限制再啟用。
API 模式：使用遠程 Embedding API，無需本地資源。
q4 適合大多數設備；q8 精度更高但體積翻倍。
只能填寫 Embedding（嵌入/向量）模型！
不要填 LLM 聊天模型（如 gemini 3、gpt-4 等），否則會報錯。
推薦：text-embedding-v3（Qwen）、text-embedding-3-small（OpenAI）
OpenAI 相容格式，會自動拼接 /embeddings 路徑。
例，填：https://dashscope.aliyuncs.com/compatible-mode/v1
填好地址和密鑰後點擊 拉取。
推薦: text-embedding-v3 (Qwen) / text-embedding-3-small (OpenAI)
適用於高維度 API 模型（如 Qwen3-Embedding、text-embedding-3-large 等）。
開啟後將關閉關鍵詞擴展、事件詞組配對、高頻詞懲罰等輔助啟發式，
完全依賴模型自身的語義理解能力進行召回，減少低品質噪音。
如果你使用的是小型或低維度模型，請不要開啟，否則召回品質會下降。
向量召回後，再用 Rerank 模型對候選結果做精細排序，顯著提升召回精度，
把真正相關的排到前面，不太相關的壓下去，讓 AI 收到的記憶更準確。
需要額外配置一個 Rerank 模型的 API（和 Embedding 模型不同），
推薦模型：Qwen3-Reranker、bge-reranker-v2-m3、bce-reranker
填好地址和密鑰後點擊 拉取。
推薦: gte-rerank (Qwen) / jina-reranker-v2-base-multilingual (Jina)
預設 Rerank 使用摘要進行排序。開啟後將使用原始訊息全文進行精排，
可以捕捉到摘要遺漏的細節，但對模型的上下文長度要求很高。
僅適用於長上下文 Rerank 模型（如 Qwen3-Reranker 系列，支援 32K）。
bge-reranker-v2-m3（≤1024 有效）和 bce-reranker（≤512）等短上下文模型
請勿開啟，全文會被截斷，效果反而更差。
每次發送訊息時，從向量庫召回最相關的 K 條記憶注入上下文。
相似度低於此值的記憶不會被召回。提高此值可減少噪音。
排名前 N 且達到全文閾值的召回結果會發送原始正文而非摘要。設為 0 關閉此功能。
只有相似度達到此值的高置信度結果才會發送全文，避免不相關內容浪費 Token。
